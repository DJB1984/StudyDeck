# Intent

Adds optional, non-gating cloud persistence — email magic-link login backed by Supabase (Postgres + Auth) — so a user's deck history and flashcard progress survive clearing browser data, switching browsers, or moving to a new device. It exists because decks are often the product of a real AI conversation, not a copy-paste, so losing one to a wiped `localStorage` is a real loss (`PRD-auth.md` Problem). Guest/logged-out use of StudyDeck must remain completely unchanged — login is a pure upgrade layered on top of the existing localStorage-only app, never a requirement to use it. Source-of-truth docs for this feature: `PRD-auth.md`, `design-doc-auth.md`, `task-list-auth.md` (repo root).

- R9 [verify: unit] [caution: session-persistence invariant] No app code sets a session expiry, forces periodic re-authentication, or clears the Supabase session on its own initiative. The app relies entirely on the Supabase client's default `localStorage`-persisted session/refresh-token behavior for "stay logged in until explicit logout" (`PRD-auth.md`: "Session persists indefinitely"). Do not add an idle-timeout or scheduled force-logout — that directly contradicts this requirement.
- R10 [verify: unit] [caution: guest-mode-unaffected invariant — the single most important behavioral guarantee of this entire feature] A user who has never logged in (no Supabase session has ever existed in this browser) experiences the app as byte-for-byte identical to pre-auth StudyDeck: identical synchronous `Storage` reads/writes (see `Storage.spec.md` R13), and no network call to Supabase at all beyond the one harmless `getSession()` check the app makes at startup to determine that there is, in fact, no session. `AuthButton`'s mere presence in the header is the only visible difference for a guest.
- R11 [verify: unit] [caution: never-block-login-on-migration invariant] `migration.ts`'s `syncOnLogin()` runs automatically on every successful login, not just the first. If the bulk write to `decks`/`flash_state` fails or partially fails (network drop, quota), the login itself still succeeds and the user's local data is left completely untouched (see R12/R14 for why the merge model makes this true unconditionally, not just as a retry policy). Sync failure surfaces, at most, a soft non-blocking toast — it must never block, delay, or roll back the login itself (`design-doc-auth.md` Error Handling: "Never block login on migration succeeding").
- R12 [verify: unit] [caution: this replaced an earlier "only upload if the account's cloud tables are completely empty" design — that version had a real bug: a deck created locally while logged out was silently discarded the moment you logged into an account that already had OTHER decks in it, because it skipped upload entirely and replaced local history with the (unrelated) cloud snapshot. Found live, 2026-07-20. Do not reintroduce an empty-check gate on upload.] `syncOnLogin()` UNCONDITIONALLY runs `upload()` on every login — every local deck/pile is upserted (by title) to the account, regardless of whether that account already has other decks. This is what makes a locally-created deck get added to an existing account instead of clobbered by it.
- R13 [verify: unit] `migration.ts` does NOT clear or modify `localStorage` during upload or merge-down — local copies remain in place as the working cache for the rest of that logged-in session. [superseded in part by R21: this no longer implies data survives past logout — see R21 for the current, revised logout behavior; this R13 still governs sync itself, which must never clear local data mid-session.]
- R14 [verify: unit] [caution: ADDITIVE merge, not a replace — this is the second half of the fix described in R12. The old design replaced local history with the cloud snapshot wholesale, which is exactly how a locally-created deck got lost. Never change `mergeDown()` back to `Storage.replaceHistory(cloudDecks)` on its own.] On every successful login, after `upload()` (R12) has run, a merge-down step reads the account's full `decks`/`flash_state` rows via `SupabaseClient` and ADDS any deck/pile whose title is not already present in the local cache — titles that already exist locally are left exactly as they are, never overwritten by the cloud copy. The write goes through Storage's normal write path (`Storage.replaceHistory`/`replaceFlashState`) so `Storage.spec.md` R17's subscribe callbacks fire. Combined with R12's unconditional upload, this makes PRD-auth.md's success criterion true in both directions: a cleared-browser login recovers everything from the cloud (nothing local to preserve, so merge-down effectively repopulates all of it), AND a deck created locally before logging into an account with existing decks is preserved AND added to that account, never overwritten by it.
- R15 [verify: unit] After migration and/or hydration (R11–R14) complete, `Storage.subscribe()`'s registered callbacks (`Storage.spec.md` R17) are invoked so already-mounted screens (Home, Flashcard) re-read Storage and reflect the synced state without requiring a manual refresh or a page reload.
- R16 [verify: manual] [caution: RLS boundary — the single most important security property of this feature; do not treat it as "just a Postgres detail" the app code can compensate for] All cloud reads/writes use the Supabase anon key from the client and rely ENTIRELY on Postgres Row Level Security policies (`user_id = auth.uid()`) on both `decks` and `flash_state` to stop one user's browser from reading or writing another user's rows. Client-side filtering after an unrestricted fetch is never a substitute for RLS — if RLS is misconfigured or disabled, the anon key can read/write every row for every user regardless of what the app's own queries look like. Must be verified with two separate real test accounts (`design-doc-auth.md`/`task-list-auth.md` Phase 1: "a hard gate, do not proceed... until this passes") before this feature is considered safe to run live.
- R17 [verify: manual] `SupabaseClient.ts` is the only module in the codebase that imports `@supabase/supabase-js` or references a Supabase client instance. `AuthButton.tsx`, `LoginModal.tsx`, `migration.ts`, and `Storage.ts` all reach Supabase exclusively through `SupabaseClient.ts`'s exported functions — mirrors the existing "`Storage.ts` is the only module that touches `localStorage`" rule in the root `CLAUDE.md`.
- R18 [verify: unit] `SupabaseClient.ts` exposes a synchronous `isLoggedIn(): boolean` (and/or `getUserId(): string | null`) getter backed by an in-memory value it keeps current via its own internal `onAuthStateChange` subscription, established once at module load. This is what lets `Storage.ts` branch on auth state (`Storage.spec.md` R13/R18) without itself becoming async or importing the Supabase client.
- R19 [verify: manual] [caution: this template lives in the Supabase dashboard's email settings, not in this repo — there is no source file to point to when verifying it; check the dashboard configuration, not the codebase] The magic-link email is sent via Supabase's custom SMTP (Resend) using a template matching the app's dark/purple identity: explicit `#0a0a0f` background (not reliant on email-client dark-mode inversion), no blur/glass/gradient effects (most email clients strip `backdrop-filter`), a solid `#7c3aed` CTA button with a plain-text fallback link beneath it, a "Didn't request this? Safe to ignore, no account changes were made." trust line, 560px max width, single column, mobile-safe tap target.
- R20 [verify: manual] The Supabase project URL and anon public key are read from Vite env vars (e.g. `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`), never hardcoded in source. The `.env` file holding them is gitignored, matching the repo's existing convention of keeping deployment/service secrets (e.g. `deploy.config.json`) out of git.
- R21 [verify: unit] [caution: this reverses R13's original framing — Davis's explicit call, 2026-07-19, after seeing the pre-R21 behavior live: shared-device privacy (the next person to open the browser shouldn't see your decks) outweighs "never strand the user," since cloud data isn't actually at risk either way] `AuthButton`'s logout handler calls `SupabaseClient.signOut()` and then `Storage.clearLocal()` — every local deck and flash pile is wiped, returning the browser to a clean guest slate. Cloud data is untouched; logging back in restores it via the normal R14 hydration pull. A guest-mode deck created *while logged out* that was never mirrored to the cloud is not specially preserved across a subsequent login (same last-write-wins limitation as R12 — not a new one).
- R22 [verify: unit] [caution: this was a real shipped bug, not a hypothetical — found via code review after Davis reported "I log back in and the deck doesn't appear" even though the cloud write had actually succeeded] `Storage.subscribe()` (`Storage.spec.md` R17) MUST have at least one real consumer or R14/R15's hydration is silently inert: the local cache gets correctly overwritten but no mounted screen re-renders to show it until an unrelated action or a hard reload happens to re-read `Storage`. `HomeScreen` and `FlashcardScreen` both call `Storage.subscribe()` on mount (unsubscribing on unmount) and re-read their state on notification — this is the fix, and it must not regress by a future refactor quietly dropping the subscription.
