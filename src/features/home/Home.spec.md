# Intent

Home is the landing screen and the entry point for every study session. It lets a student re-open any previously loaded deck instantly, add new decks by drag-drop, file picker, or pasted text, remove decks they no longer want, and — because StudyDeck ships no built-in AI — get a study set made by any external AI. The screen must be understandable at a glance by a non-technical first-time user: the tagline says what the app does, and the get-started path is short, plain-English, and free of developer jargon.

# Requirements

- R1 [verify: ui] Home renders a grid of file cards, one per entry in `Storage.getHistory()`, each showing the deck `title` and a metadata line `{count} questions · {lastOpened}`.
- R2 [verify: ui] Cards appear newest-first, matching the order Storage returns (Storage prepends new/updated entries). [caution: ordering comes from Storage's saveFile prepend behavior — Home must render history in the array order it receives, not re-sort it.]
- R3 [verify: ui] Each card has a delete control. Clicking it removes that deck via `Storage.deleteFile(title)` and re-renders the grid, WITHOUT navigating away from Home. [caution: the delete click must not also trigger the card's open action — in the legacy app this is `stopPropagation`; in React, stop the event from bubbling to the card's click handler.]
- R4 [verify: ui] Clicking a card (anywhere but the delete control) updates that entry's `lastOpened` to today, re-saves it via `Storage.saveFile`, refreshes the grid, and navigates to Mode Select for that deck.
- R5 [verify: ui] A drop zone (the add-deck surface) accepts a dragged file; on drop, the file is routed into the load flow. Dragging over the zone applies a visible highlight state that clears on drag-leave or drop.
- R6 [verify: ui] The drop handler rejects any file whose name does not end in `.json` (case-insensitive) with the error `Please drop a .json file.` and does not attempt to parse it.
- R7 [verify: ui] A "Load file" button opens a native file picker restricted to `.json`; choosing a file routes it into the same load flow as drop. [caution: after a pick, reset the input value so re-selecting the same file fires the change event again.]
- R8 [verify: ui] The load flow reads the file text, `JSON.parse`s it, and on parse failure shows the error `Invalid JSON: {message}` without adding anything to history.
- R9 [verify: ui] On successful parse, the file is validated via DeckValidation; if any errors are returned, they are shown joined by newlines and the file is NOT saved. Only a clean validation proceeds.
- R10 [verify: ui] On a valid deck, a history entry `{ name, title, count: questions.length, lastOpened: today, data: parsedDeck }` is saved via `Storage.saveFile`, the deck becomes the current file, the grid re-renders, and the app navigates to Mode Select.
- R11 [verify: ui] When history is empty, the get-started card (R12) renders as the hero — above the add-deck surface — and there is NO separate empty-state message. The card itself is the empty state. [caution: do not reintroduce a "no study sets loaded yet" sentence alongside the card; the redesign deliberately removed the duplicate copy — the card alone carries the first-run guidance.]
- R12 [verify: ui] The get-started card is headed "Make a study set with any AI" (or near-identical plain-English copy) and contains exactly 3 plain-English steps — (1) copy the prompt, (2) paste it into any AI along with your notes, (3) paste the AI's reply back here — and ONE primary button labeled "Copy Prompt". The words "format spec" and "JSON" must NOT appear anywhere in the card's user-facing text. [caution: only the button LABEL is plain-English — the copied payload is still the full `studydeck-format-spec.md` content via the `FORMAT_SPEC_MD` `?raw` build-time import (works offline, no fetch, no hand-maintained copy to drift). Do not swap in a shortened "friendly" prompt payload.]
- R13 [verify: ui] Errors surface through the shared toast (see the app-level error toast), which is dismissible and supports multi-line text.
- R14 [verify: ui] The subtitle under the "StudyDeck" heading reads "Use ⟨AI⟩ to turn your notes into ⟨output⟩" (final phrasing may vary slightly, but it must describe what the app does) with TWO rotating word slots: the AI slot cycles through ChatGPT, Claude, Gemini, Grok; the output slot cycles through flashcards, quizzes, tests. The currently displayed rotating word renders in the accent color (`--accent-light`, the token the app uses for accent-colored text).
- R15 [verify: ui] The two slots rotate on independent, staggered timers so they never flip at the same moment; each word change animates as a vertical roll of roughly 250–300ms. [caution: the stagger is deliberate — a single shared timer that flips both words simultaneously reads as a jarring full-sentence swap and must not be "simplified" back in.]
- R16 [verify: manual] Under `prefers-reduced-motion: reduce`, rotating words still swap on their timers but without the roll animation (instant text change, no translate/transition).
- R17 [verify: manual] The rotating word lists are defined at the `RotatingWord` call site (in `HomeScreen`), not buried inside the component, so adding a word is a one-line edit to an array literal.
- R18 [verify: ui] The add-deck surface carries a "Paste study set" control alongside drag-drop and "Load file". Activating it expands an inline paste box (textarea + Add + Cancel); Cancel collapses the box and discards the text with no other side effects.
- R19 [verify: unit] `stripCodeFences(text)` in `src/lib/deckText.ts` trims surrounding whitespace, then strips a leading fence line (``` ``` ``` with any optional language tag, e.g. `json`) and a matching trailing ``` ``` ``` fence line when both are present; input without a surrounding fence is returned unchanged apart from the trim. Covered here rather than by its own spec: it is a single pure helper existing solely to feed Home's paste flow.
- R20 [verify: ui] Clicking Add runs the pasted text through `stripCodeFences`, then through the SAME load flow as files: `JSON.parse` failure shows the `Invalid JSON: {message}` toast; DeckValidation errors are shown joined by newlines with nothing saved; a valid deck is saved to history and the app navigates to Mode Select. [caution: paste MUST reuse the exact same parse/validate/save path as file loading — a second divergent implementation is how the two paths silently drift apart. If you find yourself writing a second JSON.parse-and-validate block, stop.] Paste entries have no source filename, so the history entry's `name` is derived as `{title}.json` — this satisfies R10's entry shape.
- R21 [verify: unit] Both the file-read path (drop and picker) and the paste path funnel through one shared `loadDeckFromText(text, sourceName)`-style function that owns parse → validate → save → navigate; neither path duplicates that logic.
- R22 [verify: ui] When history is non-empty, the layout order is: deck grid first, then the add-deck surface, then a single compact row (copy like "Need a new set? [Copy Prompt]") in place of the full get-started card. The compact row's button copies the same `FORMAT_SPEC_MD` payload with the same "Copied!" feedback.
