# Intent

Storage is the single choke point for every localStorage read and write in StudyDeck. It exists so that no other module ever touches `localStorage` directly, which keeps persistence logic (quota handling, key naming, atomic deletes) in one place and makes a future migration to IndexedDB a one-module change rather than an app-wide rewrite. Since the auth feature (`src/features/auth/Auth.spec.md`), Storage additionally owns the guest-vs-logged-in branch point: its public surface stays synchronous and localStorage-backed no matter what, so every existing caller keeps working unmodified, while a logged-in session gets its writes mirrored to Supabase in the background and its local cache refreshed from Supabase at login time. Storage itself still never imports or references Supabase directly — that stays `SupabaseClient.ts`'s exclusive job.

# Requirements

- R1 [verify: unit] `get(key)` reads the raw string from localStorage and returns `JSON.parse` of it, or `null` if the key is absent. Any thrown error (missing key, corrupt JSON) is caught and results in `null`, never a throw.
- R2 [verify: unit] `set(key, val)` `JSON.stringify`s `val` and writes it via `localStorage.setItem`, returning `true` on success and `false` on unrecoverable failure. It never throws out to callers.
- R3 [verify: unit] [caution: quota eviction is the reason set() is more than a one-liner — do not "simplify" it back to a bare setItem. On `QuotaExceededError`, if history is non-empty, evict the OLDEST history entry (see R9 for which end that is) via `deleteFile`, surface a user-facing warning toast with the exact text "Storage full — oldest file removed to make room.", then retry the write EXACTLY ONCE. If the retry also throws, return `false`. If history was already empty, return `false` without evicting.]
- R4 [verify: unit] `getHistory()` returns the parsed `studydeck_history` array, or an empty array `[]` when the key is absent or unreadable — never `null`.
- R5 [verify: unit] `saveFile(entry)` upserts by `title`: if an existing history entry has the same `title`, it is replaced in place (position preserved); otherwise the new entry is prepended so it becomes the newest. It persists via `set('studydeck_history', ...)`.
- R6 [verify: unit] [caution: this delete MUST be atomic across BOTH keys — removing only the history entry leaves an orphaned flash-pile key that accumulates forever. Do not split these into two separately-callable operations.] `deleteFile(title)` removes the matching history entry (filtered by `title`) AND removes the `studydeck_flash_{title}` key in the same call. The flash-key removal is wrapped so a failure there cannot throw.
- R7 [verify: unit] `getFlashState(title)` returns the parsed `studydeck_flash_{title}` object, or the default `{ known: [], learning: [] }` when absent.
- R8 [verify: unit] `setFlashState(title, state)` persists `state` under `studydeck_flash_{title}` via `set` (and therefore inherits R3 quota handling).
- R9 [verify: unit] [caution: "oldest" = the LAST element of the history array, because saveFile prepends new entries (newest-first ordering). Evicting `history[0]` would delete the most recent file — the exact opposite of intended behavior.] Quota eviction in R3 removes `history[history.length - 1]`.
- R10 [verify: unit] The only localStorage keys Storage reads or writes are `studydeck_history` and `studydeck_flash_{title}`. Flash piles are stored as arrays of question `id` strings, never array indices, so pile state survives question reordering.
- R11 [verify: manual] No module other than Storage accesses `localStorage`, `window.localStorage`, or an equivalent web-storage API directly. All persistence flows through Storage's public methods. [caution: this is the whole reason the module exists — a stray `localStorage.setItem` elsewhere defeats the IndexedDB-migration abstraction and bypasses quota handling.]
- R12 [verify: manual] Storage surfaces the quota warning by calling the app's shared toast/notification mechanism, not by throwing or by console-only logging, so the user actually sees why a file disappeared.
- R13 [verify: unit] [caution: guest-mode-unaffected invariant — the single most important behavioral guarantee auth adds to this module] When no Supabase session is active (`SupabaseClient.isLoggedIn()` is `false`), every Storage method's behavior, return value, and timing is IDENTICAL to pre-auth behavior (R1–R12) — the auth-aware branches in R14/R16 simply do not execute. This must be true by construction (an `isLoggedIn()` check gating the extra work), not by coincidence of Supabase being unreachable.
- R14 [verify: unit] [caution: never let a Supabase mirror failure roll back, block, or change the synchronous return value of the local write — local write success is always the source of truth for what the caller sees] When a session IS active, `set()`, `saveFile()`, `deleteFile()`, and `setFlashState()` perform their existing localStorage write exactly as before (R2/R3/R5/R6/R8, unchanged, synchronous, same return value), and additionally fire a best-effort ASYNC mirror of the same write to Supabase via `SupabaseClient` (`saveDeck`/`deleteDeck`/`setFlashState`). This mirror is not awaited by the caller. A mirror failure is caught and logged, and surfaces at most a single non-blocking toast (e.g. "Sync to your account failed — saved locally, will retry next login."); it is never thrown and never undoes the local write.
- R15 [verify: unit] `deleteFile()`'s Supabase mirror (R14) deletes the matching row from `decks` by `(user_id, title)` and the matching row from `flash_state` by `(user_id, deck_title)` — preserving R6's atomic-across-both-keys spirit in the cloud copy too. A failure deleting one of the two cloud rows does not roll back the other or the local delete.
- R16 [verify: manual] [caution: async/sync boundary — read this in full before touching any Storage read method] `get()`, `getHistory()`, and `getFlashState()` are NEVER made to await a network call and NEVER change return type to a `Promise`, logged in or not — they always return synchronously from the local `localStorage` cache, exactly as R1/R4/R7 already specify, word for word. Cross-device sync is achieved entirely on the WRITE side (R14) plus the login-time hydration pull owned by `Auth.spec.md`, never by making reads async. Do NOT "fix" a stale-read complaint by making these methods return Promises — that breaks every existing synchronous caller across the app (e.g. `useState(() => Storage.getHistory())` in `HomeScreen`, `flashEngine.start()`'s `Storage.getFlashState()` call). The correct fix for staleness is ensuring hydration ran and called `Storage.subscribe`'s callbacks (R17) before the stale caller re-reads.
- R17 [verify: unit] [caution: this must NOT fire on every ordinary write — only on bulk external overwrites of the local cache — or every routine `saveFile`/`setFlashState` call anywhere in the app would trigger redundant re-renders app-wide] `Storage.subscribe(callback: () => void): () => void` registers a callback invoked whenever Storage's local cache is bulk-overwritten by something other than the normal in-app write path — concretely: after a login-time hydration pull and after first-login migration completes (both owned by `Auth.spec.md`). Returns an unsubscribe function. Modeled on the existing `src/lib/toast.ts` listener-bus pattern rather than introducing a new one. This is the seam screens use to know "re-read Storage, your in-memory copy may now be stale."
- R18 [verify: manual] Storage has zero import of `@supabase/supabase-js` and holds no reference to a Supabase client instance; every cloud call goes through `SupabaseClient.ts`'s exported functions, and the auth-state check in R13 goes through a synchronous getter `SupabaseClient.ts` exposes (`isLoggedIn()` or equivalent) — preserving the "each module owns its domain exclusively" rule from the root `CLAUDE.md`.
- R19 [verify: unit] `Storage.clearLocal(): void` removes every `studydeck_history`/`studydeck_flash_*` key actually present in `localStorage` (scanned by key name, not just entries listed in the current history array, so nothing is orphaned) and calls the same subscriber-notify path as R17. Called by `AuthButton`'s logout handler (`Auth.spec.md` R21) — cloud data is untouched, only the local cache is wiped.

# Change log

- 2026-07-19 (later same day): Added R19, `Storage.clearLocal()` — logout now wipes the local cache (Davis's explicit call after seeing the old "local survives logout" behavior live; see `Auth.spec.md` R21 for the reversed rationale).
- 2026-07-19: Extended for the auth feature (before-pass; see `src/features/auth/Auth.spec.md`, `PRD-auth.md`, `design-doc-auth.md`, `task-list-auth.md`). Added R13–R18: guest mode stays byte-for-byte unaffected (R13); logged-in writes keep their existing synchronous local behavior and gain a best-effort async Supabase mirror that can never block or roll back the local write (R14/R15); reads stay permanently synchronous and localStorage-backed, never becoming `Promise`-returning, even when logged in (R16 — flagged explicitly because `design-doc-auth.md`'s "delegates to SupabaseClient.ts... exact signature... zero changes to other modules" phrasing is only achievable this way; a literal always-delegate-reads-to-Supabase reading of that sentence would force every existing synchronous caller across the app to become async, which the source doc does not actually want); a new `Storage.subscribe()` bus (R17, modeled on the existing `toast.ts` pattern) lets already-mounted screens know to re-read after a login-time hydration pull or migration; Storage remains forbidden from importing Supabase directly (R18). This resolution was made to unblock before-pass spec-writing without a design meeting — flagged to Davis/main-thread in the requirements-writer summary as the one place this before-pass made a real architectural call rather than just transcribing the source docs, in case a different tradeoff is preferred before implementation starts.
- 2026-07-09: Spec authored for the React + TypeScript migration, backfilled from the legacy single-file `Storage` module in `legacy/studydeck.html`. Captures the get/set/getHistory/saveFile/deleteFile/getFlashState/setFlashState surface, atomic two-key delete, quota-eviction-with-retry, newest-first ordering, and the id-keyed flash-pile invariant. Authored as the target the React `Storage` lib is built against.
